公司簡介模組優化

LLM 語氣校正 · 誤擋教訓 · 多層防護設計

1️⃣ 問題：LLM 預設是 sales 角度，不是招募角度

公司簡介應該讓求職者了解團隊文化與工作環境，但 LLM 的預設輸出是 sales-facing：

❌ LLM 預設輸出
「我們致力於為客戶提供卓越的服務與解決方案。」
✅ 我們要的
「我們的團隊專注於技術創新，提供穩定的工作環境與成長機會。」

原因是 LLM 沒有「我現在是在招募人才」的 context awareness。

2️⃣ 解法：多層防護，不靠單一關卡

設計原則：從源頭到輸出，每一層都設防線。上一層沒擋住的，下一層補上。QualityGate 只是最後一道保險，不是守門員。

📝 源頭控管
Prompt ↓
角色定義「你是 HR，不是業務」，例句全面校正為招募角度，讓 LLM 從源頭就走對方向
📚 句型庫
Structure ↓
開場白與句型模板統一為團隊文化、工作環境、發展機會等招募語料
🔁 備援路徑
Fallback ↓
LLM 掛掉或搜尋失敗時，回退的 mock 文案也是招募語氣，不走 sales pitch
🤖 LLM ↓
生成公司簡介
🛡️ 產出攔截
QualityGate ↓
phrase 掃描殘留 sales 用語，抓到就 retry 重新生成，直到通過或達上限
🎨 後處理
Diversifier ↓
限制內容多樣化模組不要將文案放大為銷售修辭
📄 最終產出
確保招募語氣的公司簡介
核心邏輯：Prompt 先把角色設對（你是 HR，不是業務），讓 LLM 從源頭就走對方向，QualityGate 只負責攔截殘留的漏網之魚。

3️⃣ 設計過程遇到的坑

我們先在 QualityGate 加黑名單，列了 14 個 sales 字眼，包含 患者、消費者、滿足您的需求⋯⋯結果卻誤傷了 B2C 行業：

❌ 家歡復健專科診所
「我們提供患者專業的復健治療⋯⋯」
命中「患者」→ retry 5 次失敗
❌ 蜜思服飾店
「我們致力於滿足消費者對時尚的需求⋯⋯」
命中「消費者」「滿足您的需求」→ retry 9 次失敗

教訓：黑名單分不出「合法商業描述」和「銷售語氣」。診所說患者、服飾店說消費者都是事實，不是 sales pitch。14 個 phrase 砍到剩 5 個，只保留明顯 sales-pitch（為客戶提供、以客為尊 等），行業中性詞放行。

4️⃣ 結果與 Takeaways

校正前後對照：

「我們致力於為客戶提供最優質的服務，是您值得信賴的合作夥伴。」
「我們的團隊專注於技術創新，提供穩定的工作環境與多元的學習機會。」

💡 Takeaway

  • 單點防護不夠 — 黑名單會誤傷，Prompt 源頭控管才是根本
  • 多層互補 — 每一層負擔減輕，單層失效不會崩潰
  • QualityGate 是最後保險 — 精準度比數量重要，14→5 phrase

🏛️ 同場加映：抽象層設計的應用

這週另一個重點是確保每個模組都遵循同一套抽象層設計 — 新增功能時不開新路徑，而是 implement 既有的 interface。

模組目標結果
Storage 統一儲存層介面，切換 DB 不影響業務邏輯 DynamoDB → RDS 兩天上線，StorageInterface 不變
Error Handler 收斂 4 處分散的 keyword match 到統一入口 ErrorClassifier 統一分類，新增錯誤只需加 rule
Search 搜尋引擎可抽換，不影響上游流程 Tavily / Gemini / 並行搜尋 — 皆 implement SearchTool
LLM LLM 後端可替換，不影響生成邏輯 Gemini / Bedrock Nova — 皆 implement LLMProvider
